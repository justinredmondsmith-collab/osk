# Osk 1.0.0 Threat and Privacy Review

This document is a release-focused threat and privacy review for the bounded
`1.0.0` launch defined in
[`1.0.0-definition.md`](./1.0.0-definition.md). It is grounded in the current
repository state on `main`, not in the full end-state design.

## Scope

In scope:

- one coordinator-managed Linux host running the local Osk runtime
- the current FastAPI hub, local dashboard, CLI, and WebSocket member path
- cookie-backed member join/runtime sessions
- local operator and dashboard bootstrap/session files
- tmpfs runtime storage and LUKS-backed preserved evidence handling
- the current Chromium-only member-browser launch boundary

Out of scope:

- CI, GitHub Actions, and branch-only automation not present in `main`
- speculative future auth or deployment designs
- broad Firefox, Safari, or iOS compatibility claims
- guarantees beyond the current implementation and validated release boundary

## System Summary

The current release target is a local-first coordination system with a Linux
coordinator host, local FastAPI hub, member browser clients, local operator
sessions, and optional local service dependencies for the current development
runtime. The coordinator host manages runtime state under `~/.config/osk` and
`~/.local/state/osk`, serves `/join`, `/member`, `/coordinator`, REST APIs, and
`/ws`, and can invoke privileged host operations for tmpfs, LUKS, and hotspot
management.

## High-Value Assets

- shared operation join token and join URL
- local operator bootstrap and session files
- dashboard bootstrap code and dashboard session cookie
- member join and runtime cookies
- preserved evidence contents and exported archives
- runtime data in tmpfs and local state files
- TLS certificate and key material
- audit trail and wipe-follow-up history
- local coordinator host integrity

## Trust Boundaries

### 1. Member browser ↔ local hub

Entry points:

- `/join`
- `/member`
- `/api/member/session`
- `/api/member/runtime-session`
- `/ws`

Current controls:

- join token is exchanged into an `HttpOnly` cookie rather than left in the
  visible URL
- member runtime upgrades into a short-lived `HttpOnly` runtime cookie
- cookie settings are `SameSite=Strict`; `Secure` depends on HTTPS request
  scheme
- payload limits and bounded outbox/buffer settings exist in config

Key release risk:

- the trust boundary still begins with a shared per-operation join token, so a
  leaked or relayed QR/token still enables unauthorized joins until rotation

### 2. Local operator/browser ↔ coordinator admin surfaces

Entry points:

- local operator session header/bearer handling
- `/coordinator`
- `/api/operator/dashboard-session`
- `/api/coordinator/*`
- `/api/intelligence/*`

Current controls:

- short-lived local bootstrap files and session files under `~/.local/state/osk`
- one-time dashboard bootstrap code exchanged into a short-lived dashboard
  cookie
- local admin validation accepts coordinator token, operator session, or
  dashboard session

Key release risk:

- compromise of the coordinator host or local browser session compromises the
  entire admin plane

### 3. Hub process ↔ privileged host operations

Entry points:

- `sudo mount` / `umount`
- `cryptsetup`
- `keyctl`
- `nmcli`
- Compose-compatible local service commands

Current controls:

- privileged operations are not the default for every code path; they are
  concentrated in storage/hotspot/runtime tooling
- drill and readiness surfaces exist so operators can inspect boundaries before
  triggering destructive actions

Key release risk:

- the coordinator host is a single high-trust machine; local privilege abuse or
  host compromise is catastrophic for confidentiality, integrity, and wipe
  semantics

### 4. Hub ↔ preserved evidence export and handoff

Entry points:

- `osk evidence unlock`
- `osk evidence export`
- `osk evidence verify`
- `osk evidence destroy`

Current controls:

- export writes a manifest and SHA-256 sidecar
- verify path can confirm archive integrity against embedded and sidecar data

Key release risk:

- once exported, evidence leaves the protected runtime boundary and depends on
  operator handling discipline

## Attacker Model

Realistic attacker capabilities for the bounded launch:

- a person who sees or captures the join QR/token
- a legitimately joined member who misuses information they can access
- a person on the same local network path observing or interfering with traffic
- a local attacker with access to the coordinator host account or browser
- malformed or excessive member input intended to degrade availability

Non-capabilities the release should **not** claim to defeat:

- full endpoint compromise of member devices
- full endpoint compromise of the coordinator host
- total browser/OS artifact erasure after use
- all network monitoring or physical surveillance

## Highest-Priority Threats

### 1. Shared join-token misuse

- Goal: unauthorized member access or rejoin after kick
- Impacted assets: operation confidentiality, alert integrity, operator trust
- Likelihood: medium
- Impact: high
- Priority: high

Why it matters:

- the baseline join model is intentionally low-friction and shared-token based
- the current design already acknowledges that a kicked user can rejoin until
  the token is rotated

Release posture:

- acceptable only if documented honestly and paired with explicit token-rotation
  operator guidance

### 2. Incomplete wipe on disconnected or non-current browser state

- Goal: retain recoverable member data after the operator believes wipe is done
- Impacted assets: member privacy, operator trust, safety posture
- Likelihood: medium
- Impact: high
- Priority: high

Why it matters:

- connected-browser cleanup is materially stronger than before, but the repo
  still explicitly calls out disconnected browsers, browser history, OS caches,
  and preserved evidence destruction as outside the one live message

Release posture:

- `1.0.0` must keep this as an explicit non-guarantee and require documented
  operator follow-up for unresolved members

### 3. Coordinator-host compromise

- Goal: steal tokens, sessions, evidence, audit history, or trigger destructive
  operations
- Impacted assets: effectively all release-critical assets
- Likelihood: low to medium, environment dependent
- Impact: high
- Priority: high

Why it matters:

- the coordinator host stores local session files, TLS material, runtime state,
  and controls privileged operations

Release posture:

- cannot be eliminated in the bounded launch; must be treated as a catastrophic
  trust boundary and documented accordingly

### 4. Availability degradation through media or ingest abuse

- Goal: overwhelm queues or degrade coordinator responsiveness
- Impacted assets: live situational awareness, operator usability
- Likelihood: medium
- Impact: medium
- Priority: medium

Why it matters:

- member ingest accepts audio, frame, and manual report flows over the live
  runtime path

Current mitigation evidence:

- bounded queue sizes
- payload size limits
- observer clip rate limits
- idempotency windows and receipt tracking

### 5. Evidence export leakage after archive creation

- Goal: recover preserved evidence from exported archives outside the protected
  runtime boundary
- Impacted assets: preserved evidence confidentiality
- Likelihood: medium
- Impact: medium to high
- Priority: medium

Why it matters:

- export integrity is stronger than before, but confidentiality after export is
  still mostly an operator-handling problem

## Existing Mitigations Worth Preserving

- `HttpOnly` member join/runtime cookies instead of browser-managed reconnect
  secrets in normal browser flow
- `HttpOnly` dashboard session cookie with one-time dashboard bootstrap code
- short-lived local bootstrap/session files with `0600` permissions
- payload, buffer, and queue limits in config
- duplicate-safe ingest and manual replay behavior
- wipe-readiness summaries and follow-up history instead of implying blind wipe
  success
- evidence export manifest and checksum generation
- explicit safety document that already refuses anonymity or perfect-deletion
  claims

## De-Scoped For 1.0.0 Instead Of Hardened

The bounded release explicitly de-scopes the following from the `1.0.0`
security promise instead of solving them fully:

- broad browser support outside the Chromium family
- stronger-than-shared-token join identity guarantees
- universal hotspot automation across non-NetworkManager hosts
- disconnected-device wipe completion as part of a single live-broadcast
  guarantee

## Release Conclusions

For the bounded `1.0.0` launch, the current threat posture is acceptable only
under these conditions:

1. The release continues to make **no** anonymity, perfect-deletion, or
   disconnected-wipe guarantees.
2. The release treats coordinator-host compromise as catastrophic and outside
   what the product can mitigate.
3. Shared-token join risk remains an accepted tradeoff and is documented with
   operator guidance around kick and token rotation.
4. The release validates the connected Chromium-browser wipe path on the
   supported matrix and keeps all broader wipe claims out of launch language.
5. The release does not widen browser support beyond what the validation matrix
   can actually prove.

## Release-Critical Follow-Up

- validate the connected-browser wipe path on the minimum Chromium matrix
- define and document the exact operator posture for disconnected clients
- define and document the exact operator posture for preserved-evidence
  destruction and post-export handling
- keep `SAFETY.md` and release-facing docs aligned to these limits
