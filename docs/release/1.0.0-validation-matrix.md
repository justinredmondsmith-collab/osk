# Osk 1.0.0 Minimum Validation Matrix

This document defines the minimum validation matrix for the bounded `1.0.0`
launch in [`1.0.0-definition.md`](./1.0.0-definition.md).

The goal is not broad compatibility. The goal is a narrow, repeatable release
bar for the coordinator workflows and Chromium-only member boundary that the
repo intends to support.

## Validation Principles

- Prefer one narrow matrix that stays green over a broad matrix that exists
  only in prose.
- Every release-critical workflow needs a named command or documented
  procedure.
- Every scenario should produce inspectable evidence: CLI output, run
  artifacts, or both.
- Scenarios that cannot yet be run from a repo-owned path stay open as release
  blockers.

## Supported Environment Boundary

### Coordinator host

- Linux coordinator host
- Python 3.11+
- Compose-compatible local runtime when using current local service mode
- NetworkManager/`nmcli` only when hotspot automation is in scope

### Member clients

- Chromium-class browsers only
- one dedicated Chromebook/Chromium lab path for repeatable external-browser
  regression coverage
- no Firefox support claim
- no Safari/iOS support claim

## Scenario Detail

### 1. Install and preflight

Must verify:

- install prerequisites are surfaced truthfully
- hotspot/join-host guidance is legible
- missing dependencies fail clearly

Pass condition:

- operator can tell whether the coordinator host is ready without guessing

### 2. Runtime start/stop/status

Must verify:

- runtime starts on the supported Linux host
- status surfaces return usable readiness information
- stop flow exits cleanly

Pass condition:

- operator can bring the local runtime up and down without ambiguous state

### 3. Local operator and dashboard auth

Must verify:

- operator login/bootstrap flow works
- dashboard bootstrap code becomes a dashboard session
- dashboard state and stream endpoints are usable from a local
  **Chromium-class browser on the coordinator host**

Pass condition:

- operator can reach and use the local review shell without URL-token leakage

### 4. Mocked external-browser member smoke

Must verify:

- `/join` loads
- join reaches `/member`
- offline note queues
- reconnect drains the outbox
- reload resumes the session
- live wipe clears the local member state

Pass condition:

- the dedicated Chromebook lab path stays green and remains the repo-owned
  external-browser regression baseline

### 5. Live hub member runtime on supported Chromium browser

Must verify against the actual hub runtime, not only the mocked helper:

- member join on the supported browser family
- runtime auth/session continuity
- launch-supported manual/offline flows
- wipe behavior on a connected supported browser

Pass condition:

- at least one supported Chromium browser/device path is validated against the
  real hub runtime and artifacts or explicit run notes are retained

Current gap:

- `main` still lacks a tracked repo-owned real-hub automation workflow.
- The documented manual runbook is acceptable for release evidence, but the run
  itself still must be executed and retained.

### 6. Evidence export and verify

Must verify:

- export produces archive, manifest, and checksum
- verify succeeds on the produced bundle
- wipe drill can incorporate bundle verification

Pass condition:

- operator can export and integrity-check preserved evidence without ad hoc
  steps

### 7. Wipe readiness and live wipe follow-up

Must verify:

- wipe-readiness surfaces show stale/disconnected risk before wipe
- live wipe records coverage in the audit trail
- unresolved follow-up is visible and can be closed through the documented
  workflow

Pass condition:

- operators can distinguish between connected-browser wipe success and
  unresolved cleanup boundary work

## Release Evidence Expectations

Before `1.0.0`, each required scenario must have one of the following:

- a repo-owned command or workflow with stable artifacts
- a documented manual procedure plus retained output/artifacts

The minimum acceptable evidence set is:

- saved CLI output for install/preflight/runtime/evidence checks
- Chromebook lab artifacts for the mocked external-browser smoke path
- retained notes and artifacts for at least one supported Chromium real-hub
  run
- audit output for wipe coverage and follow-up behavior

## Open Matrix Blockers

- execute the documented real-hub Chromium validation run on a clean
  release-validation operation and retain its artifacts
  Status: done, but it revealed shutdown and lifecycle blockers that keep the
  matrix open
- decide whether stale active-operation rows before fresh startup are
  acceptable for launch or require a stricter reset/fix
- fix or narrow the wipe shutdown path so browser-side wipe success and
  command-level stop success agree on the supported path
- validate evidence export/verify on a real preserved bundle rather than a host
  with no visible preserved evidence
