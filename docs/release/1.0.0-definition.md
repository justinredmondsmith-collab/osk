# Osk 1.0.0 Release Definition

This document defines the intended scope of the first truthful `1.0.0`
release. It is deliberately narrower than the full end-state design in
[`docs/specs/2026-03-21-osk-design.md`](../specs/2026-03-21-osk-design.md).

The purpose of `1.0.0` is not to declare the design complete. The purpose is
to declare a bounded, supportable, validated product slice.

## Release Intent

Osk `1.0.0` should ship as:

- a local-first coordination system run by one coordinator on a supported
  Linux laptop
- a browser-based member experience for a bounded set of supported mobile or
  desktop browsers
- a release with explicit operator workflows for install, preflight, startup,
  runtime visibility, evidence export, and wipe follow-up
- a release whose claims are limited to the behaviors the repo can validate
  and document

Osk `1.0.0` should **not** be presented as:

- the full final platform described by every planned capability in the design
  spec
- a guarantee of anonymity, endpoint safety, or perfect deletion
- a broad compatibility release across arbitrary browsers, devices, and host
  setups
- a release that solves disconnected-client cleanup or evidence destruction
  more broadly than the implementation can prove

## Proposed 1.0.0 Product Boundary

The first `1.0.0` release should cover the product that already exists in the
repo, once hardened and validated:

### Coordinator host

- one coordinator-managed Linux laptop
- local hub lifecycle via `osk install`, `osk start`, `osk status`, and
  `osk stop`
- local operator auth/session flow
- local dashboard access for coordinator review and runtime visibility
- local operations tooling for hotspot guidance, tile caching, evidence
  export/verify/destroy, and wipe drills

### Member runtime

- browser-based join flow
- member runtime shell with alerts, GPS sharing, manual notes, queued replay,
  and current in-scope observer/sensor media flows
- bounded installable/offline-capable PWA behavior for previously loaded
  join/member pages
- cookie-backed runtime session continuity for reload and reconnect

### Intelligence and review

- live ingest from the currently supported member shell
- persisted observations, events, sitreps, findings, and review actions
- the current release-quality synthesis path, whether that remains the
  heuristic path with documented limits or a narrowly improved successor

### Operator workflows

- install readiness and preflight
- operation startup and local join-host guidance
- local review and runtime inspection
- evidence export plus verification
- wipe readiness review, live wipe trigger, and follow-up handling

## Proposed Launch Claims

If `1.0.0` ships, the project should be able to truthfully claim only the
following kinds of statements:

- Osk provides a local-first coordinator and member workflow on a defined
  supported environment.
- The release includes a validated minimum browser/device matrix rather than
  open-ended browser support.
- The release provides explicit operator tooling for evidence export, wipe
  readiness review, and local operational drills.
- Connected-browser wipe behavior is only claimed to the extent it has been
  exercised and documented.
- Remaining limits around disconnected clients, preserved evidence destruction,
  endpoint compromise, and operating-system artifacts are stated explicitly.

Any claim broader than that requires additional validation and should be
treated as post-`1.0.0` work.

## Minimum Supported Environment

The release still needs validation evidence, but the launch boundary for
`1.0.0` is intentionally narrow.

### Coordinator baseline

The supported coordinator environment for `1.0.0` is:

- one coordinator-managed **Linux** laptop
- Python **3.11+** runtime for the `osk` package
- a Compose-compatible local runtime when using the current local service mode
  (`docker`, `podman`, `docker-compose`, or `podman-compose`)
- `NetworkManager` / `nmcli` available if the operator intends to use the
  repo-owned hotspot commands
- local network reachability that allows member browsers to open the join URL
- `ffmpeg` in `PATH` when the operator chooses the real Whisper backend

This matches the current repo and spec boundary: Linux-only coordinator
assumptions, Compose-aware local services, and `nmcli`-based hotspot tooling.

### Member browser and device boundary

The supported member browser family for `1.0.0` is:

- **Chromium-class browsers only**

The release should explicitly support only the browser family and device
classes it can validate through repo-owned workflows. For the first release,
that means:

- Chromium-based member browsers using the current join/member shell
- one dedicated Chromebook/Chromium lab path for repeatable regression runs
- no claim of Firefox support
- no claim of Safari/iOS support
- no claim of broad “every modern mobile browser” compatibility

This is a support boundary, not proof that validation is finished. The matrix
still needs to be exercised before release, but the family boundary is now
explicitly narrow rather than open-ended.

### Required launch assumptions

- coordinator and members can reach each other over the chosen local network
- browser members run in secure/local contexts compatible with the current
  service-worker, media, storage, and cookie-backed session model
- operators do not rely on hotspot automation unless the host actually has
  NetworkManager-based control available

`1.0.0` should prefer a narrow matrix that the repo can keep green over a wide
matrix that exists only on paper.

## Workflows That Must Be Good Enough To Ship

The following workflows are on the critical path for `1.0.0`:

1. Install and preflight the coordinator host.
2. Start the local runtime and present join-host guidance truthfully.
3. Join from a supported member browser and reach the runtime shell.
4. Submit and replay at least the launch-supported manual/offline flows.
5. Surface live coordinator visibility through the dashboard and local CLI.
6. Export preserved evidence and verify the archive produced.
7. Review wipe readiness, trigger a live wipe, and complete documented
   follow-up for unresolved members.

If any of these workflows remains ambiguous, flaky, or undocumented, the
release is not ready.

## Exact Operator Workflows In Scope

`1.0.0` should explicitly promise the following operator workflows and no more:

### Setup and runtime control

- install the package and prerequisites
- run `osk doctor`
- run `osk drill install`
- start and stop the local runtime with `osk start` / `osk stop`
- inspect runtime state with `osk status`

### Operator auth and local review

- establish and inspect local operator session state
- open the local dashboard from a Chromium-class browser on the coordinator
  host
- inspect local audit, log, member, finding, and review surfaces
- use the current finding-triage and review commands that already exist in the
  repo

### Field operations tooling

- inspect and populate local tiles
- use repo-owned hotspot commands on NetworkManager hosts only, as a
  conditional host capability rather than a universal release requirement
- inspect evidence, export it, verify the resulting bundle, and destroy it when
  explicitly intended
- run the wipe drill, inspect wipe readiness, trigger a live wipe, and follow
  the documented follow-up process for unresolved members

### Member operation

- join from the supported browser family
- receive alerts and runtime state
- submit launch-supported manual reports/media
- survive reconnect/reload through the current cookie-backed runtime path
- replay queued offline items within the supported browser path

Anything beyond those workflows should be treated as post-`1.0.0` unless it is
required to defend this bounded launch.

## Explicit Non-Goals For 1.0.0

These areas should be treated as intentionally out of scope unless they become
necessary to defend the bounded launch:

- broad “works everywhere” browser support
- a fully mature coordinator dashboard beyond operator-critical workflows
- a fully polished mobile experience beyond the launch-supported shell
- a claim of comprehensive one-shot emergency wipe across disconnected devices
- a claim of broad production AI quality beyond the validated synthesis bar
- hardware-agnostic support beyond the tested coordinator baseline
- speculative architecture work that does not improve release confidence

## Accepted 1.0.0 Tradeoffs

The first release explicitly accepts these tradeoffs rather than treating them
as blockers to broad launch claims:

- the join model remains shared-token based for low-friction operation; token
  rotation is the operator response when that trust boundary is no longer
  acceptable
- coordinator-host compromise remains catastrophic and outside what Osk claims
  to mitigate
- hotspot automation is a conditional Linux/NetworkManager capability, not a
  required success path for every deployment
- disconnected-device cleanup remains a follow-up workflow, not a guaranteed
  one-shot wipe outcome

## Exit Condition For Gate 1

Gate 1 is done when:

- this release definition is accepted as the launch boundary
- the blocker checklist is the canonical source of what remains
- README, SAFETY, CONTRIBUTING, and release-facing runbooks are aligned to the
  same boundary
