# Osk 1.0.0 Synthesis Quality — Known Limits

**Date:** 2026-03-27  
**Applies to:** Release 1.0.0

---

## Summary

Osk 1.0.0 ships with a heuristic-based observation synthesizer that provides basic event detection, alert generation, and corroboration. This document describes the current capabilities and explicit limits so operators understand what the system can and cannot do.

---

## Current Synthesis Architecture (1.0.0)

**Backend:** `HeuristicObservationSynthesizer` in `src/osk/synthesis.py`

**Capabilities:**
- Pattern-based classification of intelligence observations
- Keyword-driven category assignment (police, medical, blocked routes, escalation)
- Corroboration detection when multiple members report similar incidents
- Cooldown-based event emission to prevent alert spam
- Situation report (SitRep) generation at configurable intervals
- Stateful incident tracking with severity escalation

---

## Classification Categories

| Category | Triggers | Severity Range |
|----------|----------|----------------|
| `POLICE_ACTION` | Keywords: police, officer, sheriff, trooper | ADVISORY → WARNING |
| `BLOCKED_ROUTE` | Keywords: blocked, barrier, closed, sealed | ADVISORY |
| `MEDICAL` | Keywords: medic, medical, injur, bleeding | WARNING |
| `ESCALATION` | Keywords: escalat, panic, stampede, fight | WARNING |
| `CROWD_MOVEMENT` | Location cluster detection (2+ members) | INFO → ADVISORY |
| `COMMUNITY` | Vision tags: vehicle, crowd, mounted | INFO |

## Known Limitations for 1.0.0

### 1. Keyword-Based Classification Only

**Limit:** The synthesizer uses simple keyword matching, not semantic understanding.

**Implications:**
- "Police are helping" and "Police are charging" both trigger POLICE_ACTION
- No contextual sentiment analysis
- False positives possible with ambiguous language

**Example:**
- Input: "No police presence observed"
- Result: POLICE_ACTION triggered (keyword "police" matched)

### 2. No Temporal Pattern Analysis

**Limit:** Beyond cooldown windows, the synthesizer doesn't analyze event sequences or trends.

**Implications:**
- Cannot detect "escalating tension over 10 minutes"
- No prediction of likely next events
- Each observation evaluated in isolation

### 3. Fixed Keyword Lists

**Limit:** Keywords are hardcoded in `_classify()` method.

**Implications:**
- Cannot learn new terminology
- Regional dialects or slang not recognized
- Requires code change to extend vocabulary

### 4. Limited Corroboration Logic

**Current:** Corroboration requires same deduplication key + different member.

**Limit:** Only checks for keyword overlap, not semantic similarity.

**Implications:**
- "Police advancing on north side" and "Cops moving in from north" don't corroborate
- Different descriptions of same event treated as separate incidents

### 5. No Multi-Modal Fusion

**Limit:** Audio, vision, and GPS observations are processed independently.

**Implications:**
- Audio saying "police" + vision showing police = two separate events
- GPS cluster detection doesn't incorporate member reports
- No cross-signal confidence boosting

### 6. No Historical Context

**Limit:** State expires after `incident_window_seconds` (default 180s).

**Implications:**
- Cannot reference events from earlier in the operation
- No "return of police presence" detection
- Pattern learning across operations not supported

## Configuration Tuning

Operators can adjust synthesis behavior via config:

```bash
# Prevent alert spam (default: 60s)
osk config --set synthesis_cooldown_seconds=120

# Extend incident memory (default: 180s)
osk config --set synthesis_incident_window_seconds=300

# Adjust SitRep frequency (default: 600s)
osk config --set synthesis_sitrep_interval_seconds=300
```

## Operational Guidance

### When to Trust Synthesis

✅ Good for:
- High-confidence keyword matches (e.g., explicit "medical emergency")
- Multiple members reporting similar observations (corroboration)
- Clear location-based clustering
- Real-time awareness of reported events

### When to Verify Manually

⚠️ Verify when:
- Severity seems mismatched to description
- Single-source observation with no corroboration
- Ambiguous language that could be false positive
- Contradictory reports from different members

### Post-1.0.0 Improvements Planned

- Semantic classification using local LLM
- Cross-modal signal fusion
- Temporal pattern detection
- Configurable keyword dictionaries
- Learned incident signatures

## Truthfulness Commit

Osk 1.0.0 makes **no claims** about:
- AI/ML-based semantic understanding
- Predictive intelligence
- Multi-modal fusion
- Learning or adaptation

The synthesizer is a rule-based event classifier with corroboration. Future releases will expand capabilities while maintaining explicit documentation of limits.
