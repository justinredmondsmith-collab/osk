# Osk 1.0.0 Gap Remediation Plan

**Date:** 2026-03-27  
**Status:** Requirements updated - gaps must be resolved before 1.0.0  
**Target:** 1.0.0 release with full sensor, evidence, and synthesis capabilities

---

## Executive Summary

Based on capability assessment, the following gaps must be resolved before 1.0.0:

1. **Evidence Artifact Capture** - Evidence store must contain actual audio/video, not just metadata
2. **Sensor Streaming Validation** - Must validate 5-10 concurrent sensor streams
3. **Semantic Synthesis** - Must provide AI-powered synthesis, not just keyword matching

**Estimated Timeline:** 2-3 weeks  
**Risk Level:** Medium (semantic synthesis is the largest unknown)

---

## Gap 1: Evidence Artifact Capture (HIGH PRIORITY)

### Current State
- Evidence store contains JSON metadata only
- Audio/video observations don't write source data to evidence
- Export produces ZIP with descriptions but no actual media

### Required State
- Intelligence pipeline writes audio chunks to evidence store
- Intelligence pipeline writes frame samples to evidence store
- Evidence export contains actual media files
- Configurable retention/compression policies

### Implementation Plan

**Step 1.1: Audio Artifact Storage**
- Modify `AudioIngest` to retain raw audio chunks
- Add `storage.write_evidence_artifact()` call in transcription worker
- Store with metadata linking to observation ID
- **Estimated:** 2-3 days

**Step 1.2: Frame/Video Artifact Storage**
- Modify `FrameIngest` to retain key frames
- Add storage write in vision worker
- JPEG compression at configurable quality
- **Estimated:** 2-3 days

**Step 1.3: Evidence Export Enhancement**
- Update `EvidenceManager.export()` to include media files
- Maintain manifest with SHA256 checksums for all artifacts
- Verify media integrity during `evidence verify`
- **Estimated:** 1-2 days

**Step 1.4: Retention and Cleanup**
- Configurable evidence size limit
- LRU eviction when approaching limit
- Automatic cleanup of old operations
- **Estimated:** 2 days

**Total for Gap 1:** 7-10 days

## Gap 2: Sensor Streaming Validation (HIGH PRIORITY)

### Current State
- Audio capture code exists in member.js (228 lines)
- Frame sampler exists (306 lines)
- NOT validated at 5-10 concurrent sensor scale
- Performance characteristics unknown

### Required State
- Validated with 5-10 concurrent sensor streams
- Documented CPU/battery impact on member devices
- Documented hub processing capacity
- Graceful degradation when overloaded

### Implementation Plan

**Step 2.1: Lab Validation Setup**
- Create validation script for multi-sensor testing
- Use Chromebook lab + multiple browser instances
- Generate synthetic sensor load
- **Estimated:** 2 days

**Step 2.2: Performance Baseline**
- Test with 2, 5, 8, 10 concurrent sensors
- Measure hub CPU/memory usage
- Measure processing latency
- Document bottlenecks
- **Estimated:** 3 days

**Step 2.3: Optimization (if needed)**
- Audio chunk size tuning
- Frame sampling rate adjustment
- Queue depth management
- Worker thread balancing
- **Estimated:** 3-5 days (contingent on findings)

**Step 2.4: Graceful Degradation**
- Automatic quality reduction under load
- Backpressure signaling to sensors
- Clear dashboard indicators
- **Estimated:** 2 days

**Total for Gap 2:** 10-12 days (may overlap with Gap 1)

---

## Gap 3: Semantic Synthesis (MEDIUM PRIORITY)

### Current State
- Pure keyword matching in `HeuristicObservationSynthesizer`
- Keywords hardcoded ("police", "medical", "blocked")
- No contextual understanding
- High false positive rate

### Required State
- Local LLM (Ollama) integration for synthesis
- Semantic understanding of transcripts/observations
- Context-aware classification
- Better false positive filtering

### Implementation Plan

**Step 3.1: Ollama Synthesis Adapter**
- Create `OllamaSynthesizer` class
- Prompt engineering for civilian context
- Classification by severity and category
- **Estimated:** 3-4 days

**Step 3.2: Backend Selection**
- Configurable: `heuristic` vs `ollama` backend
- Runtime switching via config
- Fallback to heuristic if Ollama unavailable
- **Estimated:** 2 days

**Step 3.3: Validation and Tuning**
- Test with real protest/demonstration scenarios
- Measure latency (target: <2s per synthesis)
- Tune prompt for accuracy
- Document limitations
- **Estimated:** 3-5 days

**Step 3.4: Integration**
- Wire into `IntelligenceService`
- Dashboard status for synthesis backend
- Config in `OskConfig`
- **Estimated:** 2 days

**Total for Gap 3:** 10-13 days

## Risk Assessment

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| Ollama synthesis too slow | Medium | High | Make heuristic the default, Ollama optional |
| 10 sensors overload hub | Medium | High | Validate at 5, document 10 as target |
| Evidence storage fills disk | Low | High | Implement size limits and rotation |
| Timeline slips | Medium | Medium | Can ship with heuristic synthesis if needed |

- [ ] Frame images written to evidence store during vision analysis

- [ ] 5 concurrent sensors validated with <5s latency
- [ ] 10 concurrent sensors tested (graceful degradation acceptable)
- [ ] Battery impact documented (<20% per hour target)
- [ ] Hub CPU usage <50% at 5 sensors

### Semantic Synthesis

- [ ] Classification accuracy >80% vs keyword baseline
- [ ] Latency <3s per synthesis
- [ ] Configurable backend (heuristic/ollama)
